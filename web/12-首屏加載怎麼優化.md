# 首屏加載怎麼優化

## 性能指標

### 1. FCP (First Content Paint)

- 衡量首次內容繪製，比LCP更早

### 2. LCP (Largest Content Paint)

- 衡量加載速度
- 合格線：2.5秒

### 3. TTI (Time To Interact)

- 可交互時間，頁面能響應用戶操作

### 4. FID (First Input Delay)

- 首次輸入延遲，衡量交互響應
- 合格線：100ms

### 5. CLS (Cumulative Layout Shift)

- 累積佈局偏移，衡量視覺穩定性
- 合格線：0.1秒

### 6. TBT (Total Blocking Time)

- 總阻塞時間，FCP到TTI之間

### 7. FP (First Paint)

- 白屏結束時間點

## 優化全鏈路，從輸入URL到首屏展示

### 階段一：網絡層優化

1. DNS預解析：<link rel="dns-prefetch" href="//api.example.com">"
2. HTTP/2多路復用：解決隊頭阻塞，同一個連接並發請求
3. CDN加速：靜態資源就近訪問
4. 預連接：<link rel="preconnect" href="//example.com">"
5. 資源預加載：<link rel="preload" as="script" href="critical.js">"
6. TLS優化：啟用TLS1.3，減少握手往返次數

### 階段二：網絡層優化

7. 代碼分割：Webpack動態import，路由級加載
8. Tree Shaking：移除未使用代碼
9. 圖片優化：WebP格式、響應式圖片 (srcset)、圖片CDN實時壓縮
10. 字體優化：font-display: swap 避免字體閃爍
11. Critical CSS內聯：首屏必需的CSS直接內聯，non-critical CSS延遲加載
12. JS異步加載：defer和async正確使用
13. 緩存策略：強緩存 (Cache-Control) + 協商緩存 (ETag)
14. Service Worker：離線緩存，二次加載秒開

### 階段二：渲染層優化

15. 減沙回流重繪：用transform代替top/left，用class批量修改格式
16. 骨架屏：提升感知性能，心理上讓用戶覺得更快
17. SSR服務端渲染：直接返回HTML，適用不常變化的頁面
18. pre-rendering：構建時生成static HTML，適用不常變化的頁面
19. CSR優化：避免巨型組件，使用`React.lazy` + `Suspense`
20. Virtual List：虛擬滾動，長列表只渲染可視區域

## SSR

### 代價

- 服務器壓力：每次請求都要重新渲染HTML，CPU開銷大
- TTFB變長：Server需要等待數據返回才能輸出HTML
- 開發複雜度：需要處理nodejs環境、window對象不存在、內存泄漏等問題
- 首屏水合 (Hydration)：客戶端需要激活事件監聽，首屏可交互時間不一定快

### 什麼時候不應該用SSR

- 頁面動態性極強，數據變化快，如股票行情
- 用戶交互複雜度高，需要大量客戶端狀態
- 首屏內容不依賴SEO，如後台管理
- 服務器資源有限，扛不住渲染壓力

### 替代方案

- 靜態站點生成 (SSG)：構建時生成HTML，適合blog、文檔
- 增量靜態生成 (ISR)：Next.js方案，靜動態結合
- CSR + 骨架屏：簡單場景足夠

## 性能監控

- 真實用戶監控 (RUM)：採集真實用戶數據
- 合成監控：LightHouse、WebPageTest 模擬測試
- 錯誤監控：Sentry、Signoz上報加載失敗資源
- 性能上報：PerformanceObserver API收集指標
