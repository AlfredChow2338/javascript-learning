# 如何優化大型列表的渲染性能

長列表的痛點不在「React 慢」，而在 **DOM 數量、主線程工作量、記憶體與網路/解碼成本** 一齊放大：一次掛上萬個節點、每次父層更新子列全重算、圖片與重計算塞滿主線程。優化順序通常是：**先減少節點與資料量（虛擬化 + 分頁）→ 再減少無效重渲染（memo / 穩定引用）→ 最後才考慮 Worker 與量測**。

### 本質一：瓶頸在哪裡

- **DOM 與排版**：節點越多，初次 mount、樣式計算、滾動與重排成本越高。
- **協調成本**：即使 Fiber 可切片，**列表子樹仍要逐項 diff**；沒有穩定 **key** 時還會誤更新（見 `2-reconciliation原理.md`）。
- **重渲染鏈**：父組件更新若讓每個列表項都重新執行 render，虛擬列表也會放大呼叫次數。
- **資料與資源**：一次載入過多筆、圖片同時請求與解碼、在主執行緒做重 filter/sort，都會拖垮體感。

### 本質二：虛擬滾動（最重要）

- 只掛 **視窗內 + 少量緩衝（overscan）** 的列，其餘用 **總高度占位**（或動態量測高度），滾動時回收/重用節點。
- **固定列高**：實作簡單、效能預測性最好；**變高列**需快取高度或量測後重算，複雜度與抖動風險較高。
- 常見庫：**react-window**（FixedSize / VariableSize / Grid）、**@tanstack/react-virtual**（hook 化、較彈性）；兩者都是「減 DOM」，不是取代 React diff 規則。

### 本質三：減少無效渲染（memo / useMemo / useCallback）

- **React.memo（列元件）**：列內容重、父常更新時，避免 props 未變仍重 render；可搭配自訂比對，但比對本身也有成本，勿過度細拆。
- **useMemo**：快取「依賴變了才算一次」的 filter / sort / 衍生陣列，避免每次 render 新建大陣列導致子列全刷。
- **useCallback**：把傳給 memo 子元件的**回呼引用穩定住**；若子元件未 memo，收益通常有限。
- **避免**：為了優化而到處包 memo，或 memo 了卻每次傳新 object / 內聯函式，反而增加雜訊與維護成本。

### 資料與載入策略

- **分頁 / 游標無限滾動**：減少「一次在記憶體與畫面上的資料量」；常與虛擬列表組合，接近底部再請求下一頁。
- **圖片**：`loading="lazy"`、適當尺寸與格式、必要時 **Intersection Observer** 再掛 src；減少同屏解碼峰值。
- **重元件列**：`React.lazy` + `Suspense` 分段載入，注意列表內大量 Suspense 邊界對體感與錯誤處理的影響。

### 主線程以外的重活

- 超大資料集的 **filter / sort / 聚合** 若造成明顯卡頓，可移到 **Web Worker**，主執行緒只接結果再餵給列表。
- 注意序列化成本與 Worker 生命週期；資料量不大時 Worker 未必划算。

### 量測與迭代

- **React DevTools Profiler**：找「誰在更新、耗時多久」；虛擬化後仍要確認列元件沒有不必要的大量 commit。
- **Performance / 長任務**：對照滾動、輸入是否被長任務堵住；先證據再優化。

### 實務取捨（簡表）

- **列數常破百、列 DOM 重**：優先虛擬滾動 + 穩定 key + 列 memo。
- **父組件很吵、列 props 很少變**：memo + useCallback 傳遞回呼。
- **資料永遠很大**：後端/協議分頁或無限載入 + 前端虛擬化；不要指望只靠 memo 扛住萬級 DOM。
- **搜尋排序很重**：useMemo 或 Worker，看瓶頸在「算」還是「畫」。

### 小結

- **問題**：DOM 多、重算多、資源一哄而上。
- **手段**：虛擬化減節點、資料分段載入、穩定 key 與選擇性 memo、重計算與必要時 Worker。
- **結果**：滾動與互動維持在可接受範圍；具體數字依裝置與列複雜度而定，以 Profiler 為準。
