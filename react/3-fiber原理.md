# React Fiber 原理

Fiber 是 React 16 起取代 Stack Reconciler 的協調架構：把整棵樹的更新拆成**可中斷、可恢復**的小單位，在堆上以**鏈表**描述樹，配合調度器讓主線程有機會處理輸入與繪製。

### 本質一：為什麼要換掉 Stack Reconciler

- 舊版用**遞迴 + 呼叫棧**做深度優先協調，一進入就必須一路跑完，**無法暫停**。
- 大樹或重更新會長時間佔住主線程，互動與動畫容易卡頓。
- Fiber 把「進行到哪一個節點」留在**堆上的物件**裡，改為**迴圈 + 單位工作**，才能配合時間切片與優先級。

### 本質二：Fiber 節點是什麼

- 每個 React element 對應一個 Fiber 節點；多個 Fiber 用指標組成**樹狀鏈表**（child / sibling / return）。
- 節點上帶類型、props、state、DOM 或組件實例、副作用標記、與**另一版本樹**的對應（alternate）等，供 diff 與提交使用。
- **遍歷順序**：大致為「先子後兄弟再回父」，對應 beginWork（向下展開）與 completeWork（向上收尾）的配合。

### 本質三：為什麼鏈表能「中斷再續跑」

- 呼叫棧上的區域狀態隨函式返回消失；Fiber 的狀態在**堆上持久存在**。
- 調度器只維護「下一個要處理的 Fiber」（next unit of work），時間到了就讓出 CPU，下次從同一進度繼續。
- 與「可中斷」配套：**渲染階段不直接依賴呼叫棧深度**，改由框架顯式保存進度。

### 雙緩存（current / work-in-progress）

- **current**：畫面上已對應的那棵 Fiber 樹。
- **work-in-progress**：本次更新正在建的那棵；未完成前不當成正式 UI，避免協調到一半使用者看到半成品。
- 兩棵樹透過 **alternate** 成對重用節點，減少配置與比對成本；完成後再切換角色。

### 兩個階段：Render 與 Commit

- **Render（協調 / reconciliation）**：依新資料建立或更新 WIP 樹、標記副作用；**可並發模式下可中斷、可丟棄重做**。
- **Commit**：把 DOM 變更、生命週期、ref 等**一次性套用**；**不可中斷**，以保畫面與內部狀態一致。

### 調度與優先級（概念）

- 更新帶優先級與過期語意；高優先級（例如使用者輸入）可打斷低優先級的協調工作。
- 實際由 **Scheduler** 與瀏覽器空閒／訊息迴圈配合，在單幀內做一小段再 yield。
- 在此之上才有 **Concurrent 模式、useTransition、useDeferredValue、Suspense** 等「不急的更新先讓路」的行為。

### 與 Reconciliation 的關係

- **Diff 規則**（同層、類型、key 等）仍是同一套協調語意；Fiber 換的是**承載與排程方式**，不是換一套比對公式。
- 可參考同目錄 `2-reconciliation原理.md`：Fiber 負責「怎麼分批做完協調」；Reconciliation 描述「比對時遵循哪些假設」。

### 小結

- **問題**：同步遞迴協調無法讓出主線程。
- **手段**：Fiber 鏈表 + 雙緩存 + 兩階段 + 調度。
- **結果**：可時間切片、可優先級、可並發特性，同時 Commit 仍保持原子與一致。
