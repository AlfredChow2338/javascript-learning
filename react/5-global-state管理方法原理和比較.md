# React 全局狀態：原理、比較與 Redux 的設計哲學

全局狀態要解決的是：**多處共享、避免深層 props 傳遞、以及更新時誰該重渲染**。不同方案在「狀態長什麼樣子」「如何改」「如何訂閱」三點上各有一套取捨；沒有 universal 最優，只有與團隊規模、可觀測性需求、心智模型是否一致。

### 本質一：為什麼會需要「庫」而不只用 Context

- **Context**：把值傳下去；預設上**值變就讓依賴該 Context 的消費者重算**（若未拆細、未 memo 化），高頻更新容易放大渲染。
- **外部 store / 原子 store**：把狀態放在 React 樹外或細粒度單元上，用**選擇性訂閱**或**依賴追蹤**，減少「不相關也重畫」。
- **結論**：問題不是「能不能用 Context 存全局」，而是「更新頻率 + 訂閱粒度 + 除錯需求」是否值得引入額外機制。

### 本質二：Redux 架構在幹什麼（機制層）

- **單一 store**：應用層級的**一個**狀態快照；讀取走 `getState`，不鼓勵隨處直接改物件。
- **唯讀 state + action**：變更只能透過帶 `type` 的**事件描述**（action）進入；**reducer** 為 `(state, action) => newState`，要求**純函數**語意（不可變更新、可預測）。
- **dispatch → 全量跑 reducer 鏈 → 通知訂閱者**：`react-redux` 的 `useSelector` 等做**選取 + 比較**，只有選到的切片變了才觸發組件更新。
- **Middleware**：在 action 抵達 reducer 前後插入橫切邏輯（日誌、thunk、異步）；本質是**可組合的 dispatch 管線**。

### 本質三：Redux 背後的設計哲學（為什麼長成這樣）

Redux 不是「為了少寫程式碼」，而是為了把**狀態變化變成可推理、可觀測、可重放的敘事**；代價是樣板與規則較多。

- **可觀測性優先於簡潔**：每一次變化都經過 **action** 這個「具名事件」，利於 log、assert、重播；與「隨便改 store 裡的欄位」相比，**因果鏈在文字上可讀**。
- **單向資料流與純 reducer**：把「發生了什麼」（action）和「狀態如何規則化地接招」（reducer）分開；**同一 state + 同一 action → 同一 next state**，利於單元測試與除錯，也方便團隊在程式碼審查時對齊語言。
- **狀態正規化與時間旅行**：序列化的 action 歷史 + 純更新，使 **Redux DevTools** 這類「時間軸」有意義；哲學上接近「把應用看成**歸約機**」，而不是一坨互相觸發的副作用。
- **與函數式／Elm 架構的血緣**：強調**明確的邊界**（邊界外是 IO，邊界內是資料變換），大型團隊可藉**慣例**降低各寫各的機率；缺點是學習成本與**小功能也要架儀式**的摩擦。
- **Redux Toolkit（RTK）的定位**：用 slice、immer、內建 thunk 等**減少樣板、收斂最佳實踐**，但**核心哲學沒變**——仍是 action / reducer / 單一 store 的敘事，只是寫起來不那麼像 2015 年的 boilerplate。

### 本質四：其他方案與 Redux 的「世界觀差異」（簡要比較）

- **Zustand**：輕量 store + `set` / 選擇器訂閱；**偏 imperative、API 面積小**，不必 Provider；適合中小型與快速迭代，**預設不強制**「每步都是可序列化事件」的紀律。
- **Jotai / Recoil**：**原子／圖**模型，細粒度依賴；適合「很多獨立小狀態 + 衍生值」與局部組合，心智接近 **spreadsheet / signal**，與 Redux 的「單一大樹 + 歸約」不同。
- **MobX**：**可變 observable + 自動追蹤**；開發體驗像 OOP/響應式，**魔法與隱性依賴**較多，與 Redux 的「顯式 dispatch 敘事」相反。
- **Context + useState/useReducer**：零依賴、簡單；**效能與結構**需自己拆 Context、memo、拆分 state，否則易踩「一改全刷」。

### 實務取捨（不追求精確數字）

- **偏規範、多人協作、要強除錯敘事 / 時間旅行**：**Redux + RTK** 仍是常見預設答案。
- **要輕、要快、域不複雜**：**Zustand** 出現率極高。
- **原子化 UI 狀態、衍生值多**：**Jotai / Recoil** 類型方案。
- **喜歡響應式、接受約定與工具鏈**：**MobX**。
- **極簡共享、更新不頻**：**Context** 或局部 state 即可。

### 小結

- **問題**：共享狀態、避免鑽孔、控制更新面積與可維護性。
- **Redux 的哲學**：用 **action + 純 reducer + 單一 store** 換 **可預測、可觀測、可團隊對齊**；成本是儀式與學習曲線。
- **其他方案**：在簡潔、原子圖、響應式之間做不同取捨；選型先看**團隊敘事是否需要與 DevTools 是否為剛需**，再看體積與 API 偏好。
