# 1.vue2和vue3的響應式有什麼區別

Vue 2用Object.definProperty遍歷data對象，給每個屬性添加getter和setter。

每個組件有一個watcher，getter裡收集依賴，setter裡通知更新。

Object.definProperty的缺點是：

1. 無法監聽數組下屬性改變
2. 對象新增屬性監聽不到，要用$set
3. 需要遞歸遍歷所有屬性，初始化性能差

### 本質一：Vue 2響應式原理 Object.defineProperty

Vue 2的響應式核心是Observer Dep Watcher三個類協作完成。

Observer負責遍歷data對象的所有屬性，對每個屬性調用Object.defineProperty，將屬性轉換為getter和setter。

在getter中，dep.target指向當前正在執行的watcher，通過dep.depend()將Watcher收集當前屬性的依賴列表，這就是依賴收集的過程。

在setter中，當屬性的值發生變化時，調用dep.notify()通知所有依賴的Watcher。

Watcher收到通知後，調用update方法，觸發組件的重新渲染。

對於對象新增屬性，Vue 2無法自動監聽，因為新增的屬性沒有經過Object.defineProperty轉換。

解決方法是用Vue.set或this.$set方法，手動將新屬性轉換為響應式。

對於數組，Vue 2無法通過下標直接監聽。因為數組的時候方法如push pop shift等是可以被攔截的，但直接修改數組下標比如arr[0] = xxx是監聽不到的。

Vue 2的解決方法是通過重寫數組的7個原型方法：push pop shify unshift splice sort reverse，當方法被調用時手動更新，同時通過遍歷數組為每個元素添加響應式。

Object.defineProperty的缺點是深度遍歷性能差。Vue 2在初始化時需要recursively遍歷data對象所有屬性，nested層級愈深，效能愈差。

### 本質二：Vue 3響應式原理 Proxy

Vue 3使用Proxy替代了Object.defineProperty。

Proxy是ES6引入的新特性，它可以創建一個代理對象，攔截對該對象的所有操作。

Proxy可以攔截多達11種操作，包括讀取屬性、設置屬性、刪除屬性、in操作符、for in循環、function調用等，全面攔截的能力解決了Vue 2的三大痛點。

第一個痛點：對象新增屬性

Proxy代理的是整個對象，無論訪問已有的屬性還是不存在的屬性，都會經過get和set的攔截。新增屬性的操作會被set攔截到，

Vue 3可以自然地將新屬性轉換為響應式，不需要像Vue 2那樣使用$set。

第二個痛點：數組下標修改

Proxy可以攔截數組下的賦值操作，同時也能攔截數組的push pop等變異方法。因為對數組調用push方法時，會先通過Proxy的get攔截拿到push函數，執行過程中修改數組長度又會觸發set攔截。

第三個痛點：初始化性能

Proxy是懶代理，只有訪問到某個屬性時才會進行攔截處理。而Object.defineProperty需要在初始化時遞歸所有屬性。

Vue 3的響應式系統對於深層嵌套的對象，只有當你真正訪問深層屬性時才會進行深度代理，初始化性能大幅提升。

Vue 3響應式的核心流程是：通過reactive函數創建一個Proxy代理對象，當訪問代理對象屬性時，觸發get攔截，進行依賴收集。當修改代理對象屬性時，觸發set攔截，執行更新。

### 本質三：ref的設計原因，為什麼還要保留ref

既然reactive能代理對象，為什麼Vue 3還要提供ref？

根本原因是Proxy無法直接代理primitive data type，number string boolean null undefined symbol。

Proxy只能代理對象，但Vue中經常要響應式的基本類型數據，如loading (boolean)和count (number)。

ref的設計就是為了解決這個問題，創建一個RefImpl類的實例，這個實例有一個私有屬性\_value。

對於基本類型，ref將其包裝成一個對象，通過value屬性的get和set來攔截讀寫。對於對象，ref內部會調用reactive將其轉換為Proxy對象。

ref在template中的自動解包機制是這樣的：在template編譯階段，Vue 3的編譯器會分析模板代碼，如果發現一個ref類型的變量，會自動添加.value操作。

在渲染函數執行時，如果訪問的是ref對象，也會自動解包。這就是為什麼在模板中寫count而不需要寫count.value的原因。

ref在響應式對象中的自動解包規則是：如果ref作為reactive對象的屬性被訪問或修改，Vue 3會自動解包，不需要寫.value。但如果ref在數組或Map中，則不會自動解包。

ref和reactive的選擇建議是，基本類型用ref，對象類型用reactive。如果不知道是什麼類型，就用ref，ref可以處理任何類型，只是訪問時需要寫.value。

### 本質四：Vue 3響應式的其他優勢 readonly和computed

除了Proxy帶來的改進，Vue 3的響應式系統還提供了readonly和computed這兩個重要API。

readonly用於創建一個只讀的響應式對象。

被readonly包的對象，無論對象是普通對象還是reactive對象，

所有屬性都變為只讀，修改只讀對象的屬性會觸發警告，但不會實際修改。

這在傳遞參數給組件或使用全局狀態時很有用，可以防止意外修改。

computed的實現原理與Vue 2類似，但Vue 3中利用了effect的副作用特性。

computed會創建一個ComputedRefImpl實例，其中緩存了計算結果。

只有當依賴的響應式數據發生變化時，computed才會重新計算。

effect是Vue 3響應式系統的核心API。

reactive ref computed本質上都是基於effect構建。

effect接受一個函數作為參數，當函數內部訪問響應式數據時，會自動收集這些數據作為依賴，當依賴變化時會重新執行這個函數。
